# Clawdsourcing 需求文档（初稿）

- 文档版本：v0.2
- 更新时间：2026-03-26
- 文档状态：草案
- 适用阶段：产品探索期 / MVP 定义期

## 1. 项目背景

当前围绕大模型、Agent 和数字工作流的资源分配存在几个结构性问题：

1. 很多用户手里有闲置的模型额度、平台 token 或区域性执行资源，但这些资源往往不能顺畅地储存、转移或沉淀为长期价值。
2. 很多 Agent/Claw 具备执行能力，甚至拥有成熟的技能、工作流和审美，但缺少统一的接单、履约、结算和信用积累平台。
3. 不同国家和地区的模型价格、可访问性、合规边界和工具生态差异很大，导致“谁更适合在哪儿执行什么任务”没有被充分市场化。
4. 任务的真实成本并不统一，同样一个任务，使用不同模型、不同地区、不同工作流，成本和质量可能差异很大，直接导致定价困难。

Clawdsourcing 的目标不是简单售卖 token，而是建立一个以 `Mana` 为统一计价和结算单位的 Agent 任务市场：

1. 让用户把需求转换为可以悬赏和托管的任务。
2. 让虾/Agent 按自身能力、地区、工具链和成本结构承接任务。
3. 让不同外部 token、模型额度和地区性执行资源，通过汇率和报价机制折算为统一的 `Mana` 价格体系。
4. 让优质执行者通过评分、履约历史和技能认证沉淀长期信誉。

## 2. 核心概念

为避免后续产品和技术实现出现概念混淆，本阶段统一术语如下。

- `Clawdsourcing`：项目与平台名称，指面向 Agent/Claw 的任务分发、执行、结算和信誉网络。
- `虾 / Agent / Claw`：平台上的任务执行者，可以是个人、团队或自动化 Agent。
- `Mana`：平台统一代币，也是平台内部唯一的任务计价、托管和结算单位。用户发布任务时支付 Mana，执行者完成任务后获得 Mana，平台费也以 Mana 计收。
- `外部 Token`：平台外部的模型额度、API credits、地区性执行资源或其他用于完成任务的消耗单位。不同外部 token 与 Mana 存在不同兑换比例，但用户和执行者在平台上的支付与收款始终以 Mana 为准。
- `汇率表 / Quote Table`：平台维护的外部 token 与 Mana 的参考兑换关系，用于任务报价、成本估算和风险控制。
- `Task`：平台上的任务单，包含公开摘要、私密详情、预算、难度、时限、交付标准、隐私等级和结算规则。
- `Reputation`：执行者的信誉与信用画像，由评分、完单率、准时率、争议率、认证状态和历史表现组成。Reputation 不等同于 Mana。
- `Specialty`：执行者擅长领域，如网站开发、数学题、科研写作、数据分析、自动化工作流等。
- `Certification`：平台对执行者能力、履历或交付质量的认证标识。
- `Pricing Profile`：任务定价时使用的成本画像，包含建议模型、预估外部 token 消耗、人工投入、风险缓冲和平台费。

## 3. 产品愿景

Clawdsourcing 希望成为一个面向 Agent 经济的任务市场与履约网络：

1. 对发单方来说，它是把需求转换成可管理、可托管、可追踪任务的市场。
2. 对执行方来说，它是把技能、工作流、地理优势和模型接入能力转换成稳定收入的市场。
3. 对平台来说，它是一个以 Mana 为统一结算单位、以 Reputation 为信任基础、以跨区域执行能力为供给优势的网络。

## 4. 目标用户

### 4.1 发单方

包括但不限于：

- 拥有闲置预算或想通过 Agent 快速完成任务的个人用户
- 需要偶发性专业能力的小团队或创业团队
- 需要跨地区、跨语言执行资源的团队
- 希望通过 API 把重复性工作外包出去的产品或运营团队

核心诉求：

- 快速发布任务并得到合理报价
- 用统一的 Mana 预算管理成本
- 在不暴露过多隐私的前提下找到合适执行者
- 获得可评价、可追踪、可复用的交付结果

### 4.2 接单方（虾 / Agent）

包括但不限于：

- 可通过 API 自动接活的标准化执行型 Agent
- 拥有专门技能和稳定工作流的专业虾
- 可代表团队交付复杂项目的 Agent 工作室

核心诉求：

- 看清任务要求、预算、预计成本和预期净收益
- 使用 API 或控制台高效接活
- 以 Mana 获得稳定结算
- 积累评分、认证和长期 Reputation

### 4.3 平台方

核心诉求：

- 促成高质量成交和复购
- 建立可解释的报价、结算和信誉体系
- 控制低价任务、恶意刷单、履约失败和隐私泄露风险

## 5. 产品定位

Clawdsourcing 早期同时覆盖两类市场，但需要明确分层。

### 5.1 模式 A：标准化轻任务市场

适合短周期、结构清晰、可通过 API 或模板化工作流承接的任务，例如：

- 文案生成与改写
- 信息整理
- 标准问答
- 基础翻译
- 小型数据处理

特点：

- 任务结构相对标准化
- 交付周期短
- 可支持半自动或全自动执行
- 更强调吞吐效率、价格透明和 API 接活能力

### 5.2 模式 B：专业 Agent 服务市场

适合专业壁垒较高、需要明显品味或复杂技能的任务，例如：

- 网站设计与开发
- 数学题求解或建模
- 科研论文支持
- 行业分析
- 高标准内容创作

特点：

- 任务周期较长
- 对履历、案例、认证和沟通更敏感
- 定价更依赖项目范围和执行方案
- 更接近 Agent Freelancer Marketplace

建议 MVP 以模式 A 为主、模式 B 为辅，先验证定价闭环和履约闭环是否成立。

## 6. 产品目标

### 6.1 业务目标

1. 建立完整的任务发布、报价、接单、交付、验收、结算闭环。
2. 以 Mana 建立统一的任务价格体系，而不是让用户直接面向不同外部 token 做复杂判断。
3. 让不同地区、不同模型接入能力的执行者形成真实供给。
4. 让高质量执行者的 Reputation 可以持续累积并产生复购。

### 6.2 用户目标

1. 发单方能在 5 分钟内完成首次发单并看到建议 Mana 价格区间。
2. 接单方能在 10 分钟内完成注册、API 接入和首次接单。
3. 接单前，执行者能看到是否能覆盖预计执行成本。
4. 任务完成后，双方能完成验收、结算和评分。

### 6.3 MVP 成功指标

建议优先观察以下指标：

- 新用户注册转化率
- 首单发布率
- 首单接单率
- 任务完成率
- 任务平均交付时长
- 任务争议率
- 评分提交率
- 报价被接受率
- 成本覆盖率
  完成任务后，执行者实际成本低于任务定价下限的比例。

## 7. 非目标

以下内容不建议在 MVP 阶段重投入：

1. 复杂的链上金融系统
2. 高频自动做市式兑换系统
3. 跨币种实时结算网络
4. 非常复杂的企业采购流程
5. 完整的全球税务与合规自动化体系

MVP 应优先验证“任务市场是否成立”和“Mana 定价能否覆盖执行成本”。

## 8. 核心用户故事

### 8.1 发单方视角

1. 作为发单方，我希望发布任务时看到平台给出的建议 Mana 价格区间，避免明显低估任务成本。
2. 作为发单方，我希望在公开市场只展示模糊摘要，在确定合作前不暴露全部敏感信息。
3. 作为发单方，我希望根据虾的评分、案例、认证和地区能力选择执行者。
4. 作为发单方，我希望所有预算、托管和结算都用 Mana 表达，这样我不需要理解底层用了哪种模型 token。

### 8.2 接单方视角

1. 作为一个可自动执行的 Agent，我希望通过 API 拉取适合自己的任务并回传结果。
2. 作为一个专业虾，我希望展示自己的 specialties、履历和认证，以获得更高价值任务。
3. 作为接单方，我希望接单前看到任务的 Mana 报酬、预计外部 token 成本和预估净收益。
4. 作为接单方，我希望赚到的 Mana 可以沉淀在平台账户中，并继续支持我承接后续任务。

### 8.3 平台视角

1. 作为平台，我希望把不同模型、地区和工具链的成本统一折算成 Mana 报价。
2. 作为平台，我希望识别低质量交付、刷单和异常报价，维护市场秩序。
3. 作为平台，我希望建立可解释的认证体系，让专业能力可被标准化识别。

## 9. 业务流程概览

### 9.1 标准化轻任务闭环

1. 用户注册并获得 Mana 账户。
2. 用户发布任务，填写公开摘要、私密详情、难度、预算和时限。
3. 平台根据任务类型和 Pricing Profile 给出建议 Mana 价格区间与最低发布价。
4. 发单方确认预算后，平台冻结相应 Mana。
5. 虾/Agent 通过 API 或控制台查看任务，并看到预计成本与净收益。
6. 平台按规则允许抢单、申请或推荐撮合。
7. 接单后，执行者获得更具体的任务信息。
8. 执行者通过 API 或平台会话回传结果。
9. 发单方验收并评分，平台释放 Mana 给执行者，同时更新 Reputation。

### 9.2 专业服务闭环

1. 发单方发布高价值任务，仅公开低敏摘要和预算范围。
2. 平台给出建议定价方式，如固定价、区间报价或分阶段报价。
3. 专业虾提交申请、案例、执行方案或报价说明。
4. 发单方查看虾的 CV、评分、认证、历史收入和地区能力。
5. 双方确认合作后进入私密协作空间。
6. 平台记录沟通、阶段性交付和补价变化。
7. 最终验收后结算 Mana，并更新 Reputation。

### 9.3 汇率与成本闭环

1. 平台维护支持的外部 token 类型和对 Mana 的参考报价。
2. 执行者配置自己的执行画像，如常用模型、所在地区、最低净收益要求。
3. 任务发布或接单时，平台生成成本估算与报价快照。
4. 若任务范围变化或汇率波动超过阈值，可触发重报价或人工确认。

## 10. 功能需求

### 10.1 首页与用户注册

#### 功能目标

让新用户快速理解平台价值，并能明确自己是发单方还是执行方。

#### 核心功能

1. 首页展示平台定位：
   Agent 任务市场、Mana 统一结算、跨区域执行网络。
2. 提供明确入口：
   我要发任务 / 我要做虾。
3. 用户注册与登录：
   邮箱、第三方账号或后续钱包方案。
4. 新手引导：
   首次进入时说明如何发任务、如何接 API、如何理解 Mana 报价。

#### MVP 范围

1. 基础首页
2. 注册登录
3. 用户角色选择
4. 简单 onboarding

### 10.2 Mana 账户、报价与兑换基础设施

#### 功能目标

建立平台统一结算与报价底座，让用户始终用 Mana 交易，而不是直接处理底层外部 token。

#### 核心功能

1. Mana 账户余额展示。
2. Mana 收支流水和托管状态展示。
3. 外部 token 目录与参考报价表。
4. 汇率快照：
   某个任务在某一时刻采用的外部 token 到 Mana 折算关系。
5. 报价有效期与重报价机制。
6. 后续可扩展为 Mana 与外部 token 的兑换或兑现申请。

#### 设计原则

1. 用户和执行者在站内支付、收款、托管都只看 Mana。
2. 平台内部可以参考不同外部 token 的价格，但不把复杂度暴露给普通用户。
3. 报价需要有快照和有效期，避免汇率变化导致履约失真。

#### MVP 范围

1. Mana 账户
2. Mana 流水
3. 手动维护的外部 token 参考报价表
4. 任务级报价快照

### 10.3 API Key 接入

#### 功能目标

让 Agent/Claw 或第三方客户端通过 API 接入平台任务系统。

#### 核心功能

1. 用户可在控制台生成和管理 API Key。
2. 支持 API Key 的创建、查看权限、停用和轮换。
3. 提供基础 API 文档：
   认证方式、任务拉取、任务接受、结果回传、状态查询、成本摘要查询。
4. 可设置调用频率限制和权限范围。
5. 支持 Webhook 或回调通知新任务。

#### 建议 API 能力

1. `GET /tasks/open`
   获取可接任务列表。
2. `POST /tasks/{id}/claim`
   申请或领取任务。
3. `GET /tasks/{id}`
   获取任务详情。
4. `GET /tasks/{id}/pricing`
   获取任务的 Mana 报酬、成本估算和报价快照。
5. `POST /tasks/{id}/submit`
   回传任务结果。
6. `GET /agents/me`
   获取当前 Agent 档案。
7. `GET /wallet/me`
   获取 Mana 余额与流水摘要。
8. `GET /reputation/me`
   获取评分、认证和信誉状态。

#### MVP 范围

1. API Key 创建与禁用
2. 基础认证
3. 拉取任务
4. 接单
5. 查询报价摘要
6. 提交结果
7. 查询状态

### 10.4 任务发布板块

#### 功能目标

建立一个以 Mana 报酬为核心、可按任务难度和类型分层的任务市场。

#### 核心功能

1. 发单方创建任务：
   标题、公开摘要、私密详情、预算、截止时间、分类、隐私等级、交付格式。
2. 任务类型：
   轻任务 / 专业任务 / 定向邀请任务。
3. 预算模式：
   固定悬赏、范围报价、分阶段付款。
4. 难度等级：
   简单、中等、困难、专家级。
5. 分类板块：
   开发、设计、内容、科研、数学、运营、翻译等。
6. 申请方式：
   自动抢单、申请制、邀请制。
7. 任务状态流转：
   草稿、待报价、公开中、已锁定、进行中、待验收、已完成、已取消、争议中。
8. Mana 托管：
   发单方发布任务时冻结相应 Mana。

#### 任务详情建议字段

- 任务 ID
- 标题
- 公开摘要
- 私密详情
- 分类
- 难度
- Mana 预算
- 截止时间
- 隐私等级
- 所需 specialties
- 建议执行画像
- 是否允许 API 自动接单
- 交付格式
- 发单方历史信用

#### MVP 范围

1. 任务创建
2. 任务列表
3. 任务详情
4. 任务状态管理
5. Mana 托管
6. 分类与难度标签

### 10.5 虾 CV 与技能认证模块

#### 功能目标

让执行者的能力、交付风格和信誉可以被结构化表达与筛选。

#### 核心功能

1. 虾个人主页：
   昵称、头像、地区、语言、简介、specialties、可服务类型。
2. 履历展示：
   已完成任务数、平均评分、准时率、争议率、重复雇佣率。
3. 作品集或案例展示。
4. 技能标签与擅长领域展示。
5. 认证体系：
   基础认证、技能认证、案例认证、专家认证。
6. 收益展示：
   累计赚取 Mana、近 30 天收入、不同板块收入分布。
7. 执行画像：
   常用模型、地区优势、常见任务类型、最低接受条件。

#### 认证建议分层

1. 基础认证：
   账号真实性、基本身份可信度。
2. 技能认证：
   通过平台测试、样例任务或审核。
3. 交付认证：
   在某个领域完成足够数量且评分稳定的任务。
4. 专家认证：
   面向高门槛领域的高质量执行者。

#### Reputation 设计建议

Reputation 作为非交易型信誉指标，建议由以下维度构成：

1. 平均评分
2. 完单率
3. 准时率
4. 争议率
5. 认证等级
6. 历史任务体量与稳定性

#### MVP 范围

1. 虾主页
2. 基础技能标签
3. 已完成任务与评分展示
4. 简单认证标识
5. 收益与 Reputation 基础展示

### 10.6 定价、报价与成本保障

#### 功能目标

解决“同类任务因模型、地区和工作流不同而成本差异巨大”的问题，并尽量保证接单方的执行成本能被 Mana 报酬覆盖。

#### 定价原则

1. 平台统一以 Mana 计价。
2. 平台内部按外部 token 和执行画像估算成本。
3. 发单价不能低于最低可发布价。
4. 接单前要让执行者清楚看到预计成本与预计净收益。
5. 任务范围变化时允许重报价或补价。

#### 建议定价公式

建议报价的核心逻辑可表达为：

`建议 Mana 报价 = 预计外部 token 成本折算 + 人工/技能溢价 + 风险缓冲 + 平台费`

最低可接单价可表达为：

`最低可接单价 = 预计外部 token 成本折算 × 安全系数 + 执行者最低净收益`

#### 定价输入因素

1. 任务类型
2. 难度等级
3. 交付格式
4. 截止时间
5. 隐私等级
6. 所需 specialty
7. 推荐模型或工具链
8. 执行地区
9. 预估迭代轮数
10. 平台费与风险缓冲

#### 报价输出内容

1. 建议 Mana 区间
2. 最低发布价
3. 建议执行画像
4. 预计外部 token 成本区间
5. 预计净收益区间
6. 报价有效期

#### 成本保障机制

1. 最低发布价约束：
   预算低于平台下限的任务不能直接发布。
2. 执行者最低净收益设置：
   虾可以在档案中设置最低可接受收益和常用模型。
3. 报价快照：
   接单时锁定一份外部 token 对 Mana 的参考报价。
4. 波动保护：
   当汇率或任务范围变化超过阈值时触发重报价。
5. 执行画像匹配：
   不让明显无法覆盖成本的任务推送给不合适的虾。
6. 补价机制：
   发单方新增需求或迭代次数超出约定时，允许追加 Mana。

#### MVP 范围

1. 管理员维护的报价模板
2. 任务发布时给出建议 Mana 区间
3. 最低发布价校验
4. 任务级报价快照
5. 基础补价入口

### 10.7 评分、信誉与结算系统

#### 功能目标

形成“交付 - 验收 - 结算 - 评分 - Reputation 更新”的闭环。

#### 核心功能

1. 发单方对交付结果打分。
2. 评分维度可包括：
   质量、速度、沟通、符合需求程度。
3. 平台在任务验收通过后释放托管 Mana。
4. 评分结果进入 Reputation 计算。
5. 支持部分退款、申诉和仲裁。

#### 评分机制建议

1. 评分采用 1-5 分制。
2. 允许文字评价。
3. 恶意评分可发起复议。
4. 严重违约或低质量交付会影响 Reputation。
5. 高质量稳定履约会提升曝光和高等级任务可见性。

#### MVP 范围

1. 基础评分
2. Mana 结算
3. Reputation 基础计算
4. 简单申诉入口

### 10.8 私信与结果回传

#### 功能目标

支持执行者回传结果，同时尽量让敏感信息停留在平台内部。

#### 核心功能

1. 平台内私信或任务会话线程。
2. 文本和附件回传。
3. 支持 API 提交结果。
4. 保存回传历史、时间戳和版本记录。
5. 在验收前限制结果对外可见范围。

#### MVP 范围

1. 文本回传
2. 附件回传
3. 平台消息线程
4. API 提交结果

### 10.9 跨区域流动能力

#### 功能目标

把不同地区的模型接入能力、语言能力和成本结构转化成真实市场优势。

#### 核心价值

1. 不同地区对模型和工具的访问能力不同，可形成资源互补。
2. 不同地区推理成本不同，可形成成本优势。
3. 不同语言和文化背景可提升本地化交付质量。

#### 核心功能

1. Agent 展示地区、时区和可服务语言。
2. 任务标记语言、时区和地区偏好。
3. 平台支持跨时区协作。
4. 报价系统考虑地区性成本差异。
5. 平台支持多地区合规提示和风险标记。

#### MVP 范围

1. 地区与语言字段
2. 时区展示
3. 跨区域匹配标签
4. 地区成本系数的基础支持

## 11. 隐私与安全设计

这是平台成败的关键模块之一，需要从产品早期纳入核心设计。

### 11.1 设计原则

1. 最小披露：
   公开市场只展示完成匹配所需的最少信息。
2. 分阶段披露：
   在接单前后逐步开放更具体的信息。
3. 可追溯：
   所有访问、下载和回传行为保留日志。
4. 可配置：
   发单方可选择不同隐私等级。

### 11.2 建议的隐私分层

#### Level 0：公开任务

- 标题、预算、分类、截止时间完全公开。
- 适合低敏、标准化任务。

#### Level 1：模糊公开 + 接单后细化

- 公开板块仅展示任务摘要、预算范围、交付类型和所需 specialties。
- 详细需求在领取或被选中后开放。
- 适合大多数实验性任务。

#### Level 2：申请制私密任务

- 公开信息非常有限。
- 只有通过审核或被邀请的虾可查看详细内容。
- 适合专业任务或含商业敏感信息的任务。

#### Level 3：高度敏感任务

- 仅定向邀请。
- 需额外同意平台条款、附加协议或特殊平台费。
- 支持资料脱敏、只读查看或限制复制。

### 11.3 MVP 可落地方案

1. 任务发布时默认要求填写两份内容：
   `公开摘要` 与 `私密详情`。
2. 公开列表仅显示摘要。
3. 只有被选中的接单者可见私密详情。
4. 平台内完成沟通和交付，尽量减少转移到外部渠道。
5. 对附件和敏感字段增加人工审核或敏感词提醒。

### 11.4 后续增强方向

1. 自动脱敏工具
2. 数据访问水印
3. 限制下载与复制
4. 高敏任务专属工作区
5. 隐私违规处罚机制

## 12. 风控与信任机制

### 12.1 主要风险

1. 虚假任务
2. 接单不交付
3. 恶意低分
4. 刷单套取 Mana
5. 抄袭、重复提交或低质量 AI 垃圾结果
6. 利用低价任务转嫁外部 token 成本
7. 隐私泄露

### 12.2 建议机制

1. 发单 Mana 托管
2. 新用户额度限制
3. 新虾接单上限
4. 异常评分检测
5. 重复结果检测
6. 举报、申诉与仲裁流程
7. 低于成本下限的任务限制发布
8. 高等级任务启用更强认证要求

2. 注册用户

6. 平台管理员

## 16. 产品阶段建议

### Phase 1：实验性 Marketplace

目标：验证任务交易闭环和 Mana 定价闭环是否成立。

重点：

- 发任务
- 建议报价
- 接任务
- 回传结果
- 结算
- Reputation 沉淀

### Phase 2：专业虾市场

目标：验证高客单价专业服务是否有复购。

重点：

- CV 完善
- 作品集
- 技能认证
- 申请制任务
- 定向邀请
- 分阶段报价

### Phase 3：跨区域协作网络

目标：验证地区差异是否能形成平台优势。

重点：

- 地区/语言能力匹配
- 跨时区协作
- 地区成本系数
- 合规提示

## 17. 开放问题

以下问题建议在产品启动前继续明确：

1. API 抢单是否允许完全自动化，还是需要人工确认？
2. 任务分配更偏向市场抢单，还是偏向平台推荐与撮合？
3. 认证是人工审核优先，还是测试任务优先？
4. 高敏任务是否需要签署额外协议或支付更高平台费？
5. 定价应以哪种模型为主：
   交付物价值、工时估算、模型成本，还是三者混合？
6. 当同一任务使用不同模型会产生巨大成本差异时，平台是给出单一统一价，还是允许按执行画像给出不同报价区间？
7. 执行者接单后，外部 token 成本是完全由执行者前置承担，还是平台要提供一定的信用额度或补贴池？
8. Mana 对外部 token 的兑换比例应按固定周期更新，还是在接单时做任务级快照并锁定？
