# Token 交易所设计草案（TokenTrader）

## 1. 是否有类似设计？

有相似思想，但通常分散在不同形态：

- **算力/带宽市场**：卖算力换代币；
- **任务众包平台**：按任务交付结算；
- **API 网关聚合**：多模型路由与成本优化；
- **积分经济系统**：平台内部积分兑换服务。

TokenTrader 的差异点是把它们组合成一个 **“LLM Token 执行力市场 + 信誉清算层”**。

## 2. 系统目标

1. 把闲置 token 配额变成可交易执行力；
2. 为高价值任务提供更稳定、更低成本的执行通道；
3. 通过统一计价和风控，降低跨模型交换摩擦；
4. 在“价格-质量-时延”三目标之间自动做最优权衡。

## 3. 角色与资产模型

### 3.1 角色
- **Requester（需求方）**：提交任务并支付预算。
- **Executor（执行方）**：消耗自己可用 token 完成任务。
- **Exchange（交易所）**：撮合、托管、风控、结算。
- **Arbiter（仲裁者，可选）**：争议仲裁。

### 3.2 资产
- **Provider Token**：具体模型供应商 token（不可直接互换）。
- **Token Credit（内部记账币）**：平台统一结算单位。
- **Reputation Score**：信誉分，影响成交与保证金比例。

## 4. 订单与撮合

### 4.1 订单结构
- 任务类型：问答、代码生成、摘要、翻译等；
- 质量等级：`economy / balanced / premium`；
- 约束：最大时延、失败重试策略、预算上限；
- 可接受模型白名单（可选）。

### 4.2 报价函数
对每个候选执行方计算：

`score = w_price * normalized_price + w_quality * quality_score + w_latency * latency_score + w_reliability * reliability_score`

> 平台选择综合得分最高的候选，而不是单纯最低价。

## 5. 风控与安全

1. **托管结算**：需求方下单时先锁定预算；
2. **执行保证金**：执行方按风险等级缴纳保证金；
3. **SLA 惩罚**：超时/失败扣减信誉与保证金；
4. **反刷单**：IP/设备指纹/行为图谱识别；
5. **配额证明**：执行方需周期性提交可用配额证明（或 API 成功率证明）。

## 6. 合规关注点

- 明确平台不是“转售官方账号”而是“任务服务撮合”；
- 检查各模型供应商 TOS 对代理调用与再分发条款；
- 用户数据最小化存储、可审计日志、PII 脱敏；
- 对高风险用途建立内容安全拦截与人工复核。

## 7. 经济模型建议（MVP）

- 交易手续费：`1%~5%`，按等级浮动；
- 奖励：高信誉执行方获得更高曝光和更低保证金；
- 惩罚：连续失败增加冷却时间与保证金系数；
- 初期不做复杂 AMM，优先走订单簿 + 评分撮合。

## 8. 技术实现建议（分期）

### 阶段 A（1~2 周）
- 单体服务：报价、下单、模拟执行；
- SQLite/Postgres 存储订单与信誉；
- 简单评分引擎 + 单元测试。

### 阶段 B（2~4 周）
- 接入真实 LLM providers；
- 任务结果评测（规则 + LLM-as-a-judge）；
- 基础风控流水线。

### 阶段 C（4~8 周）
- 多执行方竞价；
- 争议仲裁；
- 结算与审计报表；
- 可选链上记账模块。

## 9. 当前仓库原型对应关系

- `src/tokentrader/models.py`：订单/供给/报价数据结构。
- `src/tokentrader/engine.py`：评分、撮合、模拟执行。
- `src/tokentrader/service.py`：注册登录、会话、报价、执行服务层。
- `src/tokentrader/server.py`：HTTP API + 静态前端托管。
- `src/tokentrader/web/`：交易台前端页面。
